# Galilieo Atlas 内页视觉重构方案

- 状态：已完成
- 研究日期：2026-07-12
- 目标页面：关于、项目列表与详情、博客列表与文章详情、归档
- 参考站点：[XingHuiSama の 宝藏之地](https://www.xinghuisama.top/)
- 参考仓库：[heiehiehi/XinghuisamaBlogs](https://github.com/heiehiehi/XinghuisamaBlogs)
- 许可边界：上游采用 CC BY-NC 4.0；只提炼通用布局、玻璃分层、密度和响应式原则，不复制组件代码、Tailwind 类、精确色值、图片、字体组合或品牌表达。
- 完整设计记录与参考截图：[`docs/interior-pages-redesign.md`](../../docs/interior-pages-redesign.md)

## 1. 研究证据

实际截取并检查了以下日间页面的桌面与移动状态：

- `/projects`：项目矩阵、搜索框、两列玻璃卡和移动端单列。
- `/timeline`：归档标题、搜索与标签、卡片/时间线视图、桌面交错时间线和移动双列卡片。
- `/about`：封面、头像、身份标题、标签切换、正文容器和活动/评论区域。
- Galilieo 当前 `/projects/`、`/blog/`、`/archive/`、`/about/` 完整桌面页面。

从仓库核对到的主要实现事实：

- 项目列表桌面使用 `max-w-6xl`、两列网格、约 `24px` 间距，卡片约为 `bg-white/60 + backdrop-blur-xl + rounded-3xl`。
- 时间线页使用 `max-w-5xl`，搜索、标签筛选、时间线/卡片模式和大量客户端状态。
- 关于页使用 `max-w-4xl`，单一大容器、封面/头像身份锚点、标签切换和正文/活动内容。
- 文章详情使用主阅读面板和约 `320px` 侧栏。
- 上游依赖 Next.js、React、Tailwind、Framer Motion、Three.js；这些技术和交互复杂度不进入 Galilieo。

## 2. 核心判断

当前 Galilieo 内页的优点是编辑感强、真实、克制、语义清楚；主要问题是：

1. 首页已经形成雾蓝双主题玻璃系统，内页仍是几乎纯纸面，品牌连续性不足。
2. 项目和博客使用超大标题加横向条目，首屏视觉有力，但三个项目和四篇文章缺少更明确的内容视觉锚点。
3. 关于页信息真实且简洁，但像文档摘要，缺少身份入口和成长叙事。
4. 归档页正确但过于稀疏，在宽屏上没有时间流动感。

推荐方向不是复制参考站，而是：

> 将当前编辑式信息架构放进“雾海档案馆”式内页外壳：背景更克制、面板更实、排版仍以内容为主，玻璃只负责分层。

## 3. 全站内页共用系统

### 3.1 页面外壳

- Header 继续使用当前组件和导航，不复制参考站的自动隐藏、圆盘移动菜单或悬浮控件。
- 内页最大内容宽度约 `1120–1180px`，比首页 `1220px` 略窄，增强阅读聚焦。
- 当前 Demo 背景允许暂时扩展到项目、博客、归档和关于页，共用同一资源层并使用内页专用主题遮罩；授权待确认和正式上线前替换边界保持不变。
- 日间保持暖骨白与雾蓝灰，夜间保持深海蓝黑与月光蓝；不简单反色。

### 3.2 面板层级

- **标题层**：不强制放进完整卡片，保留编辑式大标题和主动留白。
- **内容层**：项目、文章、事实摘要和时间线使用半透明强面板。
- **阅读层**：文章与项目正文使用更实的阅读面板，优先保证长文对比度。
- 桌面玻璃约 `14–18px` 模糊；移动端关闭或明显降低大面积模糊。
- 圆角统一为 `20–24px`，内部媒体约 `16–20px`；不堆叠多套相近圆角。

### 3.3 明确不加入

- 当前只有三个项目和四篇公开文章，不加入搜索、筛选、标签云或卡片/时间线模式切换。
- 不加入评论、GitHub 活跃度、播放器、天气、弹幕、宠物或悬浮工具。
- 不虚构项目封面、头像、统计、经历、活动记录或文章数据。
- 不增加 React、Vue、Tailwind、Framer Motion、Three.js 或新的视觉运行时。

## 4. 项目页方案

### 4.1 桌面布局

三个项目使用等尺寸矩阵卡，不通过卡片大小区分重要性：

- 宽桌面优先三列，让三个真实项目在同一行形成完整矩阵。
- 中等视口降为两列，第三张卡保持相同尺寸，不拉伸成全宽。
- 每张卡显示真实类型、状态、角色、简介和最多 3–4 个技术标签；整卡进入详情。
- Heart Island 只在等尺寸卡内使用小范围受保护线稿，不改变 SVG，也不让其卡片变大。
- 其他项目没有正式 `cover` 时使用编号、类型和双主题 CSS 档案占位，不借用参考站素材。

### 4.2 移动端

- 三张卡按 Heart Island → AI Chat App → Galilieo Atlas 单列。
- 不采用参考站 390px 下的双列小卡；长中文标题和项目说明需要完整可读。
- 触控卡最小目标 `44px`，标签自然换行。

### 4.3 项目详情

- 详情 Header 进入较实的档案面板，突出项目名称、角色、状态和技术栈。
- 正文继续保持约 `780px` 阅读宽度，不增加无价值侧栏。
- 有正式封面时再启用媒体区；没有封面时不渲染巨大的空占位。

## 5. 博客页方案

### 5.1 当前内容规模策略

四篇文章不足以支撑搜索和筛选。保留全部静态输出，改为：

- 四篇文章参考上游归档的矩阵网格视图，使用等尺寸文章卡。
- 桌面采用两列 `2 × 2`，保证中文长标题和摘要有足够宽度；不为了填满页面压成四列。
- 每张卡显示分类、日期、标题、摘要和最多两个标签。
- 无正式 `cover` 时使用编号、分类和双主题 CSS 图形占位，不生成虚构插画。

### 5.2 移动端

- 全部文章单列，不通过卡片尺寸区分最新文章。
- 不采用参考站移动端两列文章卡；Galilieo 的标题较长，双列会损害扫描效率。

### 5.3 文章详情

- 正文放入较实的玻璃阅读面板，保留当前排版、代码块、表格、前后篇导航和无 JavaScript 阅读能力。
- 当前内容规模下不增加作者侧栏、热门文章、目录悬浮栏或评论。
- 正文背景比列表卡更实，避免背景细节干扰长文。

## 6. 归档页方案

归档参考上游时间线视图的中央主轴链路，但压缩成适合当前四篇文章的长度：

- 桌面使用中央垂直主轴，文章卡按时间顺序左右交替。
- 年/月分组、日期和节点共同表达时间流。
- 每篇文章使用紧凑玻璃卡：日期、分类、标题和短摘要；Hover/Focus 只做边框和轻微位移。
- 不复制上游数十篇内容造成的巨大纵向间隔，四篇文章保持紧凑节奏。
- 不加入搜索、标签筛选、卡片/时间线切换或封面图。

移动端将主轴移到左侧，全部卡片排列在右侧；日期与分类放在标题上方，不依赖左右交替或横向布局。

## 7. 关于页方案

### 7.1 身份锚点

借鉴参考站的“封面 + 头像 + 身份”层级，但使用 Galilieo 自有表达：

- 顶部共用当前 Demo 背景资源并使用内页专用遮罩形成横幅，不复制第三方图片。
- 使用当前 `G` 字母标识作为头像占位，不伪造真人或动漫头像。
- 标题使用 `Galilieo Atlas`，直接说明“AI 应用开发者 / 软件工程学生”。

### 7.2 内容结构

不使用标签切换，全部核心内容服务端输出：

1. Identity：身份、方向、当前状态。
2. Narrative：保留当前四段真实介绍，收敛行长。
3. Facts：当前身份、主要方向、当前项目、近期关注。
4. Timeline：只使用已确认事实形成三段成长记录：软件工程学习、Heart Island 持续开发、AI 产品开发实习。
5. Contact：复用当前真实 GitHub、邮件和 Heart Island 入口。

桌面使用约 `7:5` 的叙事 + 事实布局；移动端自然单列。页面应像个人档案，不像社交主页或互动社区。

## 8. 实施顺序（审批后）

### Phase 1：共用内页外壳

- 建立内页专用背景、宽度、面板和主题 token。
- 保留首页 token，不用内页规则反向覆盖首页。
- 验证 Header、Footer、双主题和移动导航。

### Phase 2：项目与博客列表

- 项目改为三个等尺寸卡片矩阵。
- 博客改为等尺寸 `2 × 2` 文章矩阵。
- 不改 Content Collections schema，不增加假封面。

### Phase 3：归档与关于

- 归档改为中央主轴、左右交替的紧凑时间线，移动端降为左轴单列。
- 关于改为身份档案 + 叙事 + 事实 + 真实成长时间线。

### Phase 4：详情页一致性

- 项目详情和文章详情应用较实的阅读面板与同一内页背景。
- 保持正文宽度、SEO、导航、无 JavaScript 和打印/复制可用性。

### Phase 5：验证

- 运行 `node scripts/verify.mjs` 与 `git diff --check`。
- 检查 1440×1000、1024×768、390×844 的日间和夜间。
- 检查 Reduced Motion、无 JavaScript、键盘焦点、移动菜单和横向溢出。
- 检查首页往返内页后主题、Reveal 和客户端生命周期没有重复初始化。
