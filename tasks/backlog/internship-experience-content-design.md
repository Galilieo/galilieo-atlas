# 实习经历内容与页面设计稿

- **状态：** Backlog，等待事实与公开边界确认后实施
- **目标：** 将 2026-06 至今的跨平台 AI 产品实习材料整理成“首页判断力摘要 → AI Chat App 公开案例页 → 技术博客完整复盘”的三级叙事，避免日报堆砌、内容重复和公司敏感信息泄露。

## 设计原则

1. 首页只回答“做过什么类型的问题、形成了什么判断”。
2. AI Chat App 项目页只展示允许公开的职责、方法与验证结果，不展示业务数据、内部代码、接口、路由、分支名或未公开产品细节。
3. 博客一篇只讲一个问题，按“现象 → 排查 → 根因 → 方案取舍 → 验证 → 可复用判断”组织。
4. 不编造指标；没有量化数据时使用可验证的定性结果。
5. `private: true` 不是访问控制。公开生成的 Markdown 中不能写入任何需要保密的信息。
6. MCP 目前属于学习记录，未完成真实 Server 接入前不包装成项目能力。

## 方案取舍

### 不新增 `experiences` Content Collection

当前只有一段实习经历。为它新增 collection、schema、动态路由、专用 Layout 和博客双向引用，会扩大维护面并与轻量个人站定位冲突。现阶段复用已有三层即可：

- 首页 `HeroSidePanel`、`QuickCards` 与 `AboutPreview`：扫读摘要，不新增独立 Experience 区块
- `/projects/ai-chat-app/`：经过脱敏的公开职责与案例
- `/notes/`：单个问题的完整复盘

当未来出现至少 3 段需要独立详情的正式经历时，再评估 `experiences` Collection。

### `private: true` 的语义

项目现有维护指南明确说明，`private` 不是访问控制。当前设计把它理解为“源码、产品细节和内部数据不公开”，而不是“页面不得生成”。因此 AI Chat App 可以保留经过脱敏的公开详情页，但必须满足：

- 页面显著说明非公开商业项目与公开边界；
- 只写个人职责、通用技术判断和验证结果；
- 不写内部代码、接口、业务数据、配置、未公开截图和产品路线。

如果用户确认该项目连职责摘要也不能公开，再统一过滤项目列表与 `getStaticPaths()`；不能只隐藏卡片却保留可访问路由。

## 信息架构

### 1. 首页实习摘要：保持当前固定结构

不恢复已删除的 `ExperienceSection`，也不扩大首页固定顺序。确认公开边界后，只更新 `src/data/home.ts` 中的当前经历和成长时间线，并让 Hero Side Panel、实习 Quick Card 与 About Preview 保持同一事实源。

若需要更完整的判断力叙事，优先放入 AI Chat App 项目页；首页最多保留一段短摘要，例如：

> 在跨平台 AI 产品中，我主要处理那些不能靠单个页面修好的问题：平台行为差异、复杂状态恢复、虚拟列表布局和媒体生成链路。以下只展示可公开、可验证的工作范围。

首页实习信息保持一个，不按日期拆成日报。摘要可收敛为：

> 参与基于 Ionic、Vue 3 与 Capacitor 的跨平台 AI 产品开发，围绕首页模块、系统交互、复杂表单状态、媒体生成结果和平台分支集成进行开发、排查与自测。

以下编号式 case index 改为项目详情页内容，不进入首页：

- **01 / Platform behavior**：根据 Android、iOS 与 Ionic 页面缓存机制，分别处理弹窗返回、侧滑锁定和当前页面滚动容器。
- **02 / Virtualized layout**：定位视觉高度与虚拟列表占位高度不一致造成的真机裁切，调整布局参与方式与安全高度。
- **03 / Stateful generation flow**：维护角色创建与 AI 媒体生成链路中的表单草稿、图片状态、任务结果和返回恢复。
- **04 / Integration & delivery**：按平台边界合并公共功能与原生配置，并完成静态官网结构整理和离线交付。

首页实习 Quick Card 继续链接 `/projects/ai-chat-app/`。

### 2. AI Chat App 项目页：公开案例账本

继续复用现有 `/projects/ai-chat-app/`，不新增 Experience Collection 或 CMS。

页面正文结构：

1. **公开范围说明**
   - 非公开商业项目。
   - 只展示个人职责、排查方法和已确认结果。
   - 不公开代码、接口、业务数据、内部截图和产品路线。
2. **工作范围**
   - 跨平台页面与交互开发
   - Android / iOS 行为差异
   - 复杂页面状态与媒体任务链路
   - 真机验证、构建检查与 Git 协作
3. **代表性案例**
   - 返回与弹窗不是同一种导航模型
   - 当前 tab 的真实滚动容器识别
   - 虚拟列表的视觉高度与布局高度
   - AI 生图返回后的表单草稿恢复
   - 公共功能与平台专属能力的分支集成
4. **次要交付**
   - 静态官网的结构整理、深浅色区域拆分、响应式修复与离线打包
5. **形成的方法**
   - 先确认容器、生命周期、状态所有者和平台边界，再修改表现层。
   - 视频流、普通列表和虚拟列表不强行共享同一种交互策略。
   - AI 生成问题按“输入 → 参数 → 任务 → 轮询 → 结果映射 → 展示”逐层定位。
6. **验证边界**
   - 每个案例分别标记“真机已验证 / 构建已验证 / 代码合并完成待双平台验证”。

案例采用纵向 editorial ledger：编号、问题、判断、结果四层文字；不使用虚构数据图表，不使用公司截图作为装饰。

## 视觉设计

沿用现有低饱和编辑感，不重排首页：

- 编号、平台和验证状态使用 `--mono`。
- 案例标题使用中文衬线，说明使用现有无衬线正文。
- 首页不增加 case index；项目详情的案例账本保持纵向 editorial ledger。
- 项目详情使用纵向 ledger，不使用厚重卡片、图标墙或时间轴圆点。
- “已验证 / 待验证”只做文字状态，不用绿色成功徽章。
- 没有获准使用的截图时，以排版、细线和简短流程关系表达，不生成伪造产品界面。
- 浅色与深色均复用现有语义 token，不新增独立品牌色。

## 可公开事实（当前材料已足够）

- 使用 Ionic、Vue 3、TypeScript 与 Capacitor 参与跨平台 AI 产品开发。
- 处理 Android / iOS 平台行为差异和真机问题。
- 处理 Ionic 缓存页面中的真实滚动容器选择。
- 处理虚拟列表占位高度与真实组件高度不一致导致的遮挡。
- 处理角色创建与 AI 生图 / 生视频链路中的状态连续性和结果展示。
- 参与跨分支功能集成，同时保护平台专属插件与配置。
- 整理静态官网结构、视觉分区、资源路径与离线交付包。
- 自主学习 MCP 基础概念。

## 不公开或需要泛化

- 内部分支名、commit SHA、未公开组件名、具体路由和文件路径。
- 接口地址、请求参数、taskId、远程配置、业务数据和内部日志。
- 未获许可的产品截图、Figma、代码片段和公司素材。
- 具体插件清单、权限配置和平台目录差异，除非确认可公开。
- 公司地址不进入经历案例。
- “负责”“主导”“独立完成”必须按真实协作边界使用。

## 需要用户确认

1. 正式岗位名称是“前端开发实习生”“移动端开发实习生”还是“AI 产品开发实习生”。
2. 实习准确起始日期，以及当前是否仍在职。
3. 公司全称和“AI Chat App”这一泛化称呼是否允许公开。
4. 哪些案例是独立完成，哪些是与带教或团队协作完成。
5. 7 月 9 日“AI 生图返回后草稿丢失”的真实根因、修改位置和验证结果。
6. 7 月 10 日分支合并完成后，Android 与 iOS 的最终运行验证结果。
7. 静态官网是否允许公开展示 URL、匿名截图或仅描述工作范围。
8. 是否允许公开“角色创建、AI 生图、生视频、视频 Feed”等产品能力词汇。
9. 是否有可公开的量化范围；没有就保持定性描述。

## 实施步骤

1. 用户确认公开边界和岗位信息。
2. 先更新 AI Chat App 项目正文，建立唯一的公开事实源。
3. 首页只同步该事实源的短摘要和项目入口，不复制案例索引。
4. 将高价值案例分别整理为博客草稿，不直接发布。
5. 复用现有 token 完成首页和详情页的 editorial ledger 样式。
6. 运行 `node scripts/verify.mjs`，再检查首页、项目详情、博客列表的桌面/移动端和明暗主题。

## 验收条件

- 首页 10 秒内能读出岗位、技术范围和三类核心能力。
- 项目页每个案例都有问题、判断、行动和验证边界。
- 首页、项目页与博客不复制相同长段落。
- 没有敏感信息、虚构成果或未验证指标。
- AI Chat App 项目页明确说明公开边界。
- 代表视口、明暗主题、RSS/sitemap 与构建验证通过。
